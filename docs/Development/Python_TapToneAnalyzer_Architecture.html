GuitarTap Python — TapToneAnalyzer Architecture

tap_tone_analyzer.py · 9 mixin modules · mirrors Swift TapToneAnalyzer + 8 extensions

1. Full class architecture

TapToneAnalyzer inherits from 9 mixins plus QObject. The base file holds all stored properties, __init__, Qt signals, the start() wiring method, and a handful of helper methods that Swift places directly on the class (not in any extension). Unlike Swift's @Published Combine-driven UI, Python uses Qt signals to notify the view layer.

2. FFT frame wiring and signal flow

Python replaces Swift's Combine subscriptions with Qt signal connections wired in start(). fftFrameReady (per-FFT, ~2.7 Hz) drives on_fft_frame which handles guitar-mode tap detection and peak analysis. rmsLevelChanged (~43 Hz) drives _on_rms_level_changed for plate/brace tap detection and decay tracking. gatedCaptureComplete delivers gated-FFT results to finish_gated_fft_capture on the main thread.

3. Guitar mode measurement flow

on_fft_frame is the main entry point for guitar mode. It calls detect_tap (absolute threshold hysteresis) and analyze_magnitudes on every FFT frame. On a rising edge, handle_tap_detection stores the spectrum, starts decay tracking, and schedules process_multiple_taps after the capture window. Spectrum averaging runs in the linear power domain.

4. Plate / brace gated FFT flow

Plate and brace modes bypass the continuous FFT path entirely. _accumulate_gated_samples (called via raw_sample_handler on the audio thread) maintains the 200 ms pre-roll ring buffer and accumulates the 400 ms gate window. When the window fills, it emits gatedCaptureComplete (queued Qt connection) delivering the samples to finish_gated_fft_capture on the main thread. The phase-state machine progresses through L → C → FLC with Accept/Redo at each review step.

5. Control, calibration, and device management

ControlMixin owns the lifecycle: start_tap_sequence, cancel_tap_sequence, reset, pause/resume_tap_detection, and the Accept/Redo phase transitions. It also owns calibration loading and device switching. Hot-plug events arrive on the main thread via _devicesRefreshed signal, triggering PortAudio re-init and mic.load_available_input_devices().

Key architecture notes

Swift @Published → Python Qt signals

Every Swift @Published property that drives UI has a corresponding Qt signal emitted explicitly from code. For example, writing status_message always goes through _set_status_message() which emits statusMessageChanged, mirroring Swift's automatic objectWillChange trigger on every assignment. The measurementComplete, tapCountChanged, plateAnalysisComplete, and materialSpectraChanged signals map to Swift's frozen-spectrum and isMeasurementComplete publishers.

Gated capture ownership

Pre-roll buffer (_pre_roll_buf), gated accumulator (_gated_accum), _gated_capture_active, and _gated_lock live on TapToneAnalyzer, not on _FftProcessingThread. This matches Swift where TapToneAnalyzer.accumulateGatedSamples(_:sampleRate:) owns the buffers. _accumulate_gated_samples is set as mic.raw_sample_handler so it is called by the processing thread on every audio chunk. When the window fills, it emits gatedCaptureComplete (a queued Qt signal on the processing thread) which delivers the samples to finish_gated_fft_capture on the main thread.

Key Python-only differences from Swift

  • _set_status_message(msg) — explicit helper for all status string writes, emitting statusMessageChanged. Swift uses @Published var statusMessage which auto-notifies.
  • recreate_proc_thread() — destroys and replaces _FftProcessingThread. Swift achieves equivalent reset via AVAudioEngine.stop() / start().
  • _on_devices_refreshed() — calls sd._terminate() + sd._initialize() before re-enumerating. Swift CoreAudio doesn't cache the device list.
  • _update_frequency_bins() — called after file playback changes mic.rate. Mirrors Swift's updateFrequencyBins() in startFromFile.
  • Three-layer peak selection (effective_longitudinal_peak_id etc.) uses Python properties with or-chain logic, exactly mirroring Swift's ?? chain.
  • set_frozen_spectrum uses Python's GIL as the atomicity guarantee (no objectWillChange.send() needed) — both arrays update before any view read since all reads are on the main thread.

Measurement persistence differences

Python delegates to views.tap_analysis_results_view.save_all_measurements / load_all_measurements rather than owning the file URL itself (Swift's measurementsFileURL is a static property on TapToneAnalyzer). The test-path redirect to a temp directory in Swift is handled by the view layer in Python.
