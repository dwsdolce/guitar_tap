1. Full class architecture

RealtimeFFTAnalyzer inherits from two mixins. The base file holds all stored properties and __init__, plus the Python-only _FftProcessingThread (a QThread that handles off-main-thread DSP — there is no Swift equivalent). The module also re-exports all FFT functions from _fft_processing for backward compatibility.

2. Audio processing pipeline

PortAudio calls new_frame on every block, which enqueues the chunk into self.queue. _FftProcessingThread.run() drains the queue, maintains the ring buffer, computes per-chunk RMS, fires raw_sample_handler on every chunk (so TapToneAnalyzer gets the pre-roll and gated accumulator), and emits fftFrameReady for each complete FFT window. The Swift equivalent uses AVAudioEngine taps delivering directly to audioProcessingQueue.

3. FFT processing functions (_fft_processing module)

The module provides pure side-effect-free functions. dft_anal is the direct counterpart to Swift's computeFFT(on:) — both are the pure DSP core used by the test suite. Swift uses vDSP via deinterleaved split-complex format; Python uses scipy.fft with a zero-phase rotation trick. Both paths are equivalent: rectangular window for the live display path, Hann window for the gated plate/brace path.

4. Device management and hot-plug monitoring

load_available_input_devices queries PortAudio and applies the same four-priority auto-selection policy as Swift's loadAvailableInputDevicesMacOS. set_device closes and reopens the stream, auto-loads calibration via the _on_calibration_changed callback, and registers the per-device sample-rate listener. Hot-plug monitoring is platform-split: CoreAudio listener on macOS, CM_Register_Notification on Windows, pyudev on Linux. Sample-rate change detection uses a CoreAudio listener on macOS and a polling thread on Windows/Linux (no native event).

_FftProcessingThread — Python-only

Swift delivers audio on AVAudioEngine's dedicated audio thread with no separate QThread needed. Python's PortAudio callback must return immediately, so the DSP work is deferred to _FftProcessingThread. It owns the ring buffer (_input_buffer), the rolling peak history (_recent_peak_history), and the calibration lock. It does not own the pre-roll buffer or gated accumulator — those were moved to TapToneAnalyzer to match Swift's ownership model.

dft_anal vs computeFFT — implementation differences

Both are pure side-effect-free DSP cores used by the test suite. Swift uses vDSP_DFT_zrop (Accelerate) with deinterleaved split-complex format and explicit scale = 1/fftSize normalization. Python uses scipy.fft.fft with a zero-phase rotation into fftbuffer and normalizes implicitly via window / sum(window). Both apply a rectangular window for the live display path and a Hann window for the gated plate/brace path, producing equivalent output.

Signal level properties

  • rms_amp / rmsLevelChanged — instantaneous per-chunk RMS (~43 Hz). Mirrors Swift inputLevelDB. Used by plate/brace tap detection and decay tracking.
  • recent_peak_level_db — rolling max RMS over 0.5 s (thread-safe lock). Mirrors Swift recentPeakLevelDB. Used only at tap-fire time to set tapPeakLevel.
  • fft_peak_amp — instantaneous FFT peak (per-frame, ~2.7 Hz). Mirrors Swift peakMagnitude. Used by guitar-mode tap detection.

Calibration application

Swift applies calibration inside performFFT(on:) after calling computeFFT, keeping computeFFT pure. Python applies it inside _FftProcessingThread.run() after calling dft_anal, using the same pattern: mag_y_db = mag_y_db + calibration. Both mirror the same comment: "mirrors Python _FftProcessingThread.run(): mag_y_db = mag_y_db + calibration".

Platform-specific hot-plug monitoring

macOS uses AudioObjectAddPropertyListener on kAudioHardwarePropertyDevices (mirrors Swift's registerMacOSHardwareListener). Windows uses CM_Register_Notification from cfgmgr32. Linux uses pyudev (optional dependency) watching the sound subsystem. Sample-rate change detection is native on macOS (kAudioDevicePropertyNominalSampleRate listener) but requires polling every 5 s on Windows and Linux since neither platform provides a native per-device sample-rate-changed event.
